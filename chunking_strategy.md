# 청킹 전략

문서 파싱 후 검색 가능한 단위로 분할/저장하는 전략을 정의한다.

---

## 1. 핵심 방향

**문서 유형 분석 → 목차 기반 계층 분리 → 최소 분류 단위 청킹 → Auto-Merge 검색**

- 분석: 문서 유형(document_type.md)을 먼저 파악하여 문서 구성 방식 결정
- 분리: 목차 페이지가 있으면 목차 기반으로 대/중/소 분류 파악
- 청킹: 목차의 가장 작은 분류 단위로 분리, 2048토큰 초과 시 하위 분할
- 검색: Proposition 임베딩으로 검색, Auto-Merge로 동적 병합

참고: HiChunk (Tencent, 2025) — Auto-Merge retrieval algorithm

---

## 2. 인덱싱 파이프라인

```
문서 (PDF, DOCX, HWPX 등)
  │
  ▼
[1] 문서 유형 분석 (LLM)
  │  → document_type.md 기준으로 문서 유형 판별
  │  → 문서 구성 방식 결정 (계층 구조, 논문 구조, 양식 등)
  │
  ▼
[2] Docling 파싱
  │  → 문서 구조 자동 인식 (제목, 소제목, 문단, 표, 이미지)
  │  → 헤더/푸터 패턴 필터링 (반복 텍스트 제거)
  │
  ▼
[3] 목차 기반 계층 분리
  │  → 3-1. 목차 페이지 탐색 및 파싱
  │         목차가 있으면: 목차에서 대분류/중분류/소분류 추출
  │         목차가 없으면: Docling 헤딩 + LLM으로 계층 추론
  │  → 3-2. 목차의 가장 작은 분류 단위로 1차 분리
  │  → 3-3. 각 분류에 Docling 파싱 결과(문단, 표, 이미지) 배정
  │  → 3-4. parent_section_id 부여
  │
  ▼
[3-5] 섹션 단위 요약 생성 (LLM)
  │  → 목차 분리 결과(대/중/소분류)별로 2~3문장 요약 생성
  │  → section_summary로 저장, section_summary_embedding 생성
  │  → 검색 파이프라인의 목차 레벨 검색(Index Search)에 사용
  │
  ▼
[4] 토큰 크기 조정
  │  → 각 청크의 토큰 수 확인:
  │     512~2048 토큰: 그대로 1청크 ✓
  │     > 2048 토큰: 본문 내 소제목/번호 패턴으로 하위 분할
  │     < 512 토큰: 그대로 유지 (Auto-Merge가 검색 시 처리)
  │
  ▼
[5] Proposition 추출 (LLM)
  │  → 각 청크의 핵심을 하나의 명제로 요약
  │  → 모델: Qwen2.5-3B-Instruct (로컬)
  │
  ▼
[6] Contextual Chunking 접두어 생성
  │  → "[도메인: {domain_category}]
  │     [문서: {title} > 섹션: {대분류} > {중분류} > {소분류}]"
  │
  ▼
[7] 임베딩 및 저장
     → section_summary → section_summary_embedding (목차 레벨 검색용)
     → Proposition + 접두어 → proposition_embedding (청크 레벨 검색용)
     → 원본 청크 텍스트 → content (답변 생성용, 임베딩하지 않음)
```

### 목차 기반 분리 예시

```
[1] 문서 유형 분석
  → 3-4. 매뉴얼/가이드 (여러 저자의 실무 교재 합본)

[2] 목차 파싱 (4페이지)
  대분류: □기능성 섬유가공 일반, □텐터가공편, □코팅/라미네이팅 가공편, □각종 기능성 가공편
  중분류: 1. 코팅가공의 일반, 2. 투습방수소재..., 3. 직물의 coating... (총 13개)
  소분류: 목차에 없음

[3] 가장 작은 분류(중분류) 단위로 1차 분리 → 13개 청크

[4] 토큰 크기 조정
  "3. 직물의 coating 가공기술 실무" → 12,000토큰 (> 2048)
  → 본문 내 소제목으로 하위 분할:
    "3-1. 코팅 설비"     (800토큰) ✓
    "3-2. 코팅 방법"     (1,200토큰) ✓
    "3-3. 코팅 수지"     (950토큰) ✓
    ...
```

## 3. 저장 스키마 (Layer 3 — 섹션/청크 인덱스)

| 필드 | 타입 | 설명 |
|---|---|---|
| `chunk_id` | string (UUID) | 청크 식별자 |
| `document_id` | string (UUID) | 상위 문서 참조 |
| `parent_section_id` | string (UUID) | 상위 섹션 참조 (Auto-Merge용) |
| `section_name` | string | 소속 섹션명 (소제목) |
| `section_depth` | integer | 섹션 깊이 (1=대분류, 2=중분류, 3=소분류) |
| `section_path` | string | 섹션 경로 ("2. 기능성 부여가공 > 2.1 방수가공 > (1) 불통기성 방수가공") |
| `section_summary` | string | 섹션 요약 2~3문장 (LLM 생성, [3-5] 단계에서 생성) |
| `section_summary_embedding` | vector | 섹션 요약 임베딩 (목차 레벨 검색용) |
| `proposition` | string | LLM이 추출한 핵심 명제 |
| `contextual_proposition` | string | 접두어 + Proposition (임베딩 원본) |
| `proposition_embedding` | vector | contextual_proposition의 임베딩 벡터 (청크 레벨 검색용) |
| `content` | string | 원본 섹션 텍스트 (답변 생성용) |
| `token_count` | integer | 원본 텍스트 토큰 수 |
| `keywords` | list[string] | 섹션 키워드 |

## 4. 검색 파이프라인 (Auto-Merge)

```
쿼리
  │
  ▼
[1] 쿼리 임베딩 생성
  │
  ▼
[2] Proposition 임베딩 검색 (Vector Search)
  │  → Top-K 관련 섹션 검색
  │
  ▼
[3] Auto-Merge 판정
  │  검색된 섹션들의 parent_section_id 확인
  │  → 같은 상위 섹션의 하위 섹션이 2개 이상 검색됨
  │     → 해당 상위 섹션의 모든 하위 섹션을 병합하여 전달
  │  → 단독 검색된 섹션
  │     → 해당 섹션만 전달
  │
  ▼
[4] 토큰 크기 제어
  │  병합 결과가 토큰 제한을 넘는지 확인
  │  → 최소: 512 토큰 (미만이면 인접 형제 섹션 추가)
  │  → 적정: 1024 토큰
  │  → 최대: 2048 토큰 (초과 시 검색 스코어 낮은 섹션부터 제외)
  │
  ▼
[5] LLM 답변 생성
     → 병합된 섹션 텍스트 + 메타데이터 → LLM → 답변
```

### Auto-Merge 예시

```
쿼리: "불통기성 방수가공의 처리 방법"

검색 결과 (Top-5 Proposition 매칭):
  1. chunk_A: "(1) 불통기성 방수가공" (parent: "2.1 방수가공")  ← score 0.82
  2. chunk_B: "(2) 통기성 방수가공"   (parent: "2.1 방수가공")  ← score 0.71
  3. chunk_C: "2.1 방수가공"          (parent: "2. 기능성 부여가공") ← score 0.68
  4. chunk_D: "2.2 발수가공"          (parent: "2. 기능성 부여가공") ← score 0.55
  5. chunk_E: "1.1 수지가공제"        (parent: "1. 형태안정화가공")  ← score 0.42

Auto-Merge 판정:
  → chunk_A, chunk_B가 같은 parent ("2.1 방수가공")
  → "2.1 방수가공"의 모든 하위 섹션을 병합:
    (1) 불통기성 방수가공 + (2) 통기성 방수가공 + (3) 투습방수 가공
  → 병합 결과: ~290 토큰 (512 미만)
  → 상위의 chunk_C "2.1 방수가공" 본문도 포함 → ~400 토큰
  → 아직 512 미만이므로 chunk_D도 포함 가능 → ~600 토큰 ✓

LLM에 전달되는 컨텍스트:
  [2.1 방수가공 본문] + [(1) 불통기성] + [(2) 통기성] + [(3) 투습방수] + [2.2 발수가공]
  = ~600 토큰 (적정 범위)
```

## 5. Proposition 추출 전략

### 프롬프트

```
당신은 문서 분석 전문가입니다.
주어진 섹션의 핵심 내용을 하나의 명제(Proposition)로 요약하세요.
반드시 한 문장으로만 답하세요.
```

### 검증 결과 (테스트 기반, "기능성 가공 기술 실무" PDF)

| 테스트 | Proposition ↔ 하위 유사도 | 표준편차 | p-value |
|---|---|---|---|
| 문단 단위 Prop ↔ 문장 | 0.6181 | 0.1053 | 6.52e-226 |
| 섹션 단위 Prop ↔ 문단 | 0.5326 | 0.1115 | 8.97e-30 |

→ 섹션 단위에서도 통계적으로 유의미하게 대표함 (p < 0.001)
→ 문단 수 5개 이하 섹션: 유사도 0.57~0.71 (충분)
→ 문단 수 6개 이상: 유사도 하락 → 중목차 단위 청킹으로 적정 크기 유지 시 완화

### Multi-Proposition 검증

문단 5개 이상 섹션에서 Proposition 2개 추출 시:
- 평균 개선폭: +0.01 (미미)
- 개선 섹션 비율: 61.8%
- LLM 호출 2배 대비 효과 부족 → **Proposition 1개로 통일**

### 목차 기반 분리의 필요성

패턴 기반 계층 분류(depth 추정)의 한계:
- Docling 원본: 1,124개 섹션 (과잉 분할)
- 패턴 기반 중목차 추정: 248개 청크 (실제 13개 대비 19배 과잉)
- 원인: 합본형 문서에서 각 원고 내부 번호가 1부터 재시작 → 중목차와 소목차 구분 불가

→ 패턴 매칭이 아닌 **목차 페이지 파싱**으로 실제 계층을 확정해야 함
→ 목차가 없는 문서는 LLM이 Docling 헤딩 목록을 보고 계층 추론

### Contextual Chunking 접두어 포맷

```
[도메인: 산업/제조 > 산업기술·진흥]
[문서: 기능성 가공 기술 실무 > 섹션: 2. 기능성 부여가공 > 2.1 방수가공 > (1) 불통기성 방수가공]
불통기성 방수가공은 직물 표면에 고무나 합성수지를 도포하여 완전 방수 기능을 부여하는 가공법이다.
```

→ 접두어 ~30토큰 추가 (전체 대비 3~5%)
→ 인덱싱 시 1회 처리, 검색 시 추가 비용 없음

## 6. 특수 컨텐츠 처리

| 컨텐츠 타입 | 처리 방식 |
|---|---|
| **텍스트** | 위 파이프라인 적용 |
| **이미지** | VLM으로 Description 생성 → 텍스트와 동일하게 Proposition 추출 및 임베딩. 원본은 MinIO 저장 |
| **표** | Table Description 생성 → Proposition 추출 및 임베딩. Raw 데이터는 RDB 적재 |
| **수식/화학식** | 텍스트 변환 후 포함. 변환 불가 시 이미지 처리 흐름으로 전환 |

## 7. 근거

| 설계 요소 | 근거 |
|---|---|
| 섹션 단위 저장 | Financial Report Chunking (2024) — 문서 구조 기반 청킹이 고정 크기보다 성능 우수 |
| 목차 기반 분리 | 테스트 검증 — 패턴 기반 248청크 vs 목차 기반 13청크, 패턴 매칭으로는 합본형 문서 구조 파악 불가 |
| Auto-Merge | HiChunk (Tencent, 2025) — 세밀 저장 + 검색 시 동적 병합 |
| Proposition 검색 | Dense X Retrieval (2024) — 명제 단위 검색이 검색 정확도 향상 |
| Proposition 1개 | 테스트 검증 — 2개 추출 시 개선폭 +0.01로 미미, LLM 호출 대비 효과 부족 |
| Contextual Chunking | Anthropic Contextual Retrieval (2024) — 문맥 접두어로 검색 실패율 67% 감소 |
| 토큰 크기 기준 | Rethinking Chunk Size (2025), NVIDIA Benchmark — 맥락 이해형은 512~1024 최적 |
| 동적 단위 선택 | Mix-of-Granularity (COLING 2025) — 쿼리에 따라 최적 단위가 다름 |
